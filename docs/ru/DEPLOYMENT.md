> Версия: 1.1.0
> Статус: рабочая версия
> Тип документа: инструкция по установке и развёртыванию
> Язык: русский

---

## Назначение

Этот документ описывает, как развернуть Family Health OS с нуля.

Цель — получить рабочую систему из личных Google Sheets, одной семейной Google Sheets и ChatGPT-проекта, который работает по спецификациям из GitHub.

Репозиторий проекта указывается в рабочей настройке проекта:

```text
<repository_url>
```

Пример формата:

```text
https://github.com/<owner>/<repository>
```

GitHub содержит только системную часть проекта: архитектуру, инструкции, шаблоны, спецификации и промты.

Рабочие данные семьи хранятся в Google Sheets.

Персональные и медицинские данные нельзя сохранять в GitHub.

---

## Что должно получиться

После развёртывания должны быть готовы:

* Личные Google Sheets для участников семейной системы
* Одна семейная Google Sheet для общей координации
* ChatGPT-проект с подключёнными инструкциями
* Правила маршрутизации данных по таблицам и вкладкам
* Базовое оформление таблиц
* Проверка структуры

Шаблонная структура таблиц:

```text
Adult_1_health
Adult_2_health
Child_1_health
Family_health
```

Шаблонные `member_id`:

```text
adult_1
adult_2
child_1
family
```

Реальные имена участников, реальные названия таблиц и реальные `member_id` не должны храниться в публичной инструкции.

Если в приватной инструкции проекта указаны другие рабочие названия таблиц и `member_id`, нужно использовать их.

---

## Что нужно до начала

Потребуется:

* Аккаунт Google
* Доступ к Google Sheets
* ChatGPT-проект
* Доступ к репозиторию проекта
* Список участников семейной системы
* При наличии — доступ агента к Google Drive и Google Sheets

Если у агента есть доступ к Google Drive и Google Sheets, таблицы можно не создавать вручную.

Если доступа нет, таблицы создаются вручную по этой инструкции.

## Основная схема системы

```text
GitHub
  ↓
Архитектура, инструкции, правила, шаблоны
  ↓
ChatGPT-проект или агент
  ↓
Маршрутизация данных, анализ, ежедневные и недельные разборы
  ↓
Google Sheets
  ↓
Рабочие данные семьи
```

GitHub отвечает за правила системы.

Google Sheets отвечают за рабочие данные.

ChatGPT или внешний агент связывает эти два слоя: учитывает правила из репозитория, принимает данные от пользователя, определяет нужную таблицу и вкладку, помогает обновлять записи, выводы, задачи и обзоры.

## Шаг 1. Подключить репозиторий

Репозиторий проекта должен быть указан в инструкции ChatGPT-проекта:

```text
<repository_url>
```

Ассистент должен считать репозиторий главным источником правил проекта.

Если инструкции в чате противоречат содержимому репозитория, приоритет имеет репозиторий.

---

## Шаг 2. Подключить репозиторий через GitHub-коннектор

Если в рабочей среде доступен GitHub-коннектор, подключите репозиторий:

```text
<repository_full_name>
```

Пример формата:

```text
<owner>/<repository>
```

После подключения проверьте, что ассистент может учитывать файлы:

```text
README.md
docs/ARCHITECTURE.md
docs/CORE_PRINCIPLES.md
docs/ru/DEPLOYMENT.md
docs/ru/SETUP_REPOSITORY_AND_SHEETS.md
docs/USAGE.md
docs/MASTER_PROMPT.md
```

Если GitHub-коннектор недоступен, используйте ручной вариант:

1. Откройте репозиторий в браузере
2. Скопируйте содержимое `docs/MASTER_PROMPT.md`
3. Вставьте его в инструкции ChatGPT-проекта
4. Добавьте ссылку на репозиторий
5. При необходимости прикрепите ключевые `.md`-файлы к проекту

## Шаг 3. Выбрать сценарий создания таблиц

Есть два сценария.

### Сценарий А. Агент создаёт и оформляет таблицы

Используется, если у ChatGPT-проекта или внешнего агента есть доступ к Google Drive и Google Sheets.

В этом сценарии пользователь не создаёт таблицы руками.

Пользователь поручает агенту:

1. Создать рабочие Google Sheets
2. Создать все обязательные вкладки
3. Добавить заголовки колонок
4. Настроить базовое оформление
5. Проверить структуру
6. Дать отчёт о готовности

Агенту можно поручить настройку только технической структуры таблиц.

На этапе настройки агент не должен добавлять реальные персональные или медицинские данные.

### Сценарий Б. Таблицы создаются вручную

Используется, если у агента нет доступа к Google Drive и Google Sheets.

В этом сценарии пользователь создаёт таблицы и вкладки вручную, а агент помогает:

* Проверить структуру
* Подготовить заголовки
* Сформировать правила оформления
* Найти ошибки
* Обновить инструкцию

---

## Шаг 4. Создать личные таблицы

Создайте по одной Google Sheets на каждого участника.

Шаблонные названия:

```text
Adult_1_health
Adult_2_health
Child_1_health
```

Личная таблица используется для подробных данных одного человека.

Один человек — одна личная таблица.

Если в приватной инструкции проекта указаны другие рабочие названия, используйте их.

---

## Шаг 5. Создать семейную таблицу

Создайте отдельную Google Sheets:

```text
Family_health
```

Эта таблица используется только для координации:

* Общие задачи
* Статусы
* Ближайшие проверки
* Семейные решения
* Медицинские визиты
* Общие ресурсы
* Ссылки на личные таблицы

Подробные личные данные в семейной таблице не хранятся.

---

## Шаг 8. Настроить идентификаторы

Используйте стабильные `member_id`.

Шаблонные значения:

```text
adult_1
adult_2
child_1
family
```

Шаблонное соответствие:

```text
Adult_1_health → adult_1
Adult_2_health → adult_2
Child_1_health → child_1
Family_health → family
```

Один человек — один стабильный идентификатор во всех таблицах.

Для взрослого профиля:

```text
profile_type | adult
child_profile | false
```

Для детского профиля:

```text
profile_type | child
child_profile | true
```

Детский профиль нельзя интерпретировать по взрослой логике без проверки возрастного контекста.

---

## Шаг 9. Оформить таблицы

Для всех вкладок рекомендуется единый базовый формат:

* Первая строка — заголовки
* Первая строка закреплена
* Включён фильтр
* Заголовки выделены жирным
* Текст в ячейках переносится
* Даты оформлены в формате `yyyy-mm-dd`
* Время оформлено в формате `hh:mm`
* Для `confidence` используются значения `high`, `medium`, `low`
* Для `source` используются значения `user`, `photo`, `document`, `lab`, `app`, `doctor`, `assistant`, `manual`

Рекомендуемые служебные поля для регулярных записей:

```text
date
time
member_id
source
confidence
comment
```

Не каждая вкладка обязана содержать все эти поля, но для регулярных записей они должны быть предусмотрены.

## Шаг 10. Настроить ChatGPT-проект

В инструкции ChatGPT-проекта вставьте содержимое файла:

```text
docs/MASTER_PROMPT.md
```

В инструкции должна быть ссылка на репозиторий проекта:

```text
<repository_url>
```

В приватной инструкции проекта можно указать реальные рабочие значения:

```text
adult_1 → <private_member_name>
adult_2 → <private_member_name>
child_1 → <private_member_name>
family → family
```

Реальные имена, реальные ссылки на таблицы и реальные медицинские данные не должны попадать в публичные файлы репозитория.

## Шаг 13. Проверить запуск системы

Отправьте тестовое сообщение:

```text
Я: Adult_1
Тестовая запись для проверки маршрутизации
```

Ассистент должен определить:

1. Кто пишет
2. Кому принадлежат данные
3. Какая таблица нужна
4. Какая вкладка нужна
5. Какие поля должны быть заполнены
6. Нужно ли обновить аналитику или семейные задачи

Для проверки измерения можно использовать обезличенный пример:

```text
Я: Adult_1
Вес сегодня 93 кг
```

Ожидаемая маршрутизация:

```text
Таблица: Adult_1_health
Вкладка: Measurements
member_id: adult_1
source: user
confidence: high
```

Это тестовая запись. Реальные рабочие данные нужно отправлять только в приватном рабочем проекте.

---

## Шаг 14. Рабочий принцип после запуска

Каждое новое сообщение с данными должно проходить через цепочку:

```text
Данные → маршрутизация → запись → интерпретация → гипотеза → решение → рекомендация
```

Ассистент должен сначала определить, к кому относятся данные, и только потом выбирать таблицу и вкладку.

Если неясно, кто пишет или о ком данные, нужно уточнить.

---

## Шаг 15. Принцип приватности

GitHub содержит систему и документацию.

Google Sheets содержат рабочие данные.

ChatGPT помогает маршрутизировать и анализировать данные, но не должен превращать чат в основное хранилище.

Персональные и медицинские данные нельзя сохранять в GitHub.

В GitHub допустимы только:

* Пустые схемы
* Обезличенные примеры
* Инструкции
* Спецификации
* Документация

В публичной документации нельзя хранить:

* Реальные имена участников
* Реальные даты рождения
* Реальные ссылки на личные таблицы
* Реальные медицинские документы
* Реальные лабораторные результаты
* Реальные симптомы
* Реальные назначения
* Реальные фотографии
* Другие идентифицирующие данные

## Шаг 16. Что делать, если агент не может создать таблицы

Если агент сообщает, что у него нет доступа к Google Drive или Google Sheets, используйте ручной сценарий.

Попросите агента подготовить структуру для ручного переноса:

```text
У тебя нет доступа к созданию Google Sheets.

Подготовь структуру таблиц Family Health OS для ручного переноса.

Нужно выдать:

1. Список таблиц
2. Список вкладок для личных таблиц
3. Список вкладок для семейной таблицы
4. Рекомендуемые заголовки
5. Правила оформления
6. Проверочный чек-лист

Используй шаблонные названия:

Adult_1_health
Adult_2_health
Child_1_health
Family_health

Не используй реальные имена участников.
```
